## Scope Clarification: What Phase This Covers

This plan covers **Phase C** (manual validation, 5-20 leads) plus the **expansion items**
that improve Phase C before it runs. Phase A (automation) begins only after 3×30-day paying
customers are confirmed. All items below are for Phase C unless explicitly marked Phase A.

## Vision

### 10x Check
The 10x version of ALBGE is a platform where any local marketing freelancer or agency
targets a city + category, and ALBGE generates 50 pre-built sites with personalized
outreach and tracks conversions. Revenue: $99/mo per seat, platform takes 20% of setup
fees. TAM: every local marketing agency that currently cold-calls small businesses.

### Platform Potential
Core infrastructure (GMB scraper + site generator + lead state machine) is reusable.
Phase B multi-tenancy turns this into an API product. Keep architecture compatible.

## Lead State Machine

```
new → site_generated → screenshot_taken → outreach_ready
         ↓
     outreach_sent → payment_received → approved → LIVE
         ↓
     opted_out (any stage)
         ↓
     no_reply (after 7 days, no automation in Phase C)
```

State is tracked in two places:
- **Phase C:** Local CSV (`leads-tracker.csv` at project root) — source of truth
- **Phase A+:** Supabase `leads` table — source of truth

CSV → Supabase migration happens at Phase A kickoff. CSV is the Phase C log only.

## Accepted Scope (Phase C additions)

### 1. Auto-screenshot of generated site

In `generate.ts`, after `npm run build` completes:
- Spin up local HTTP server on port 8766 (fixed — no port collision at Phase C volume)
- Use `~/.claude/skills/gstack/browse/dist/browse` to screenshot at 1440×900 viewport
- Wait up to 5 seconds for page load (timeout → fallback)
- Save to `sites/[slug]/screenshot.png`
- Kill the local server after screenshot is taken
- Screenshot path referenced in outreach template and CSV

Fallback: if browse binary not found (`which browse` returns nothing), OR screenshot times
out, skip screenshot silently, log "screenshot unavailable" in console output, and continue.
No crash — missing screenshot does not block outreach template generation.
Screenshot timing: after build, before CSV row is written.

### 2. CSV lead tracker

`generate.ts` appends one row to `leads-tracker.csv` at project root after each successful
build. Format:

```
slug,business_name,phone,city,category,site_url,screenshot_path,outreach_message,status,created_at
austin-pro-plumbing,Austin Pro Plumbing,(512) 555-0192,Austin TX,Plumber,https://...,sites/austin-pro-plumbing/screenshot.png,"Hi Mike...",outreach_ready,2026-03-28T10:00:00Z
```

Duplicate prevention: check first column (`slug`) before appending. If match found, print
"⚠️  Lead [slug] already in tracker — skipping CSV write." to stdout. Idempotent.
CSV library: use Node's built-in `fs.appendFileSync` with manual CSV escaping — no
additional npm dependency needed for this simple format. One row = one `appendFileSync` call.
No concurrent write risk in Phase C (single-operator CLI, not a server).

### 3. Personalized outreach message

In `generate.ts`, after scraping, make one Claude API call to generate the first line:

```typescript
// Input: lead.businessName, lead.reviews[0] (if exists), lead.category
// Prompt: "Write ONE sentence (max 20 words) that references this specific review to
//          open a cold WhatsApp message. First name only. Direct. No fluff."
// Output: "Hi Mike, I saw Sarah's review about your emergency response — impressive."
// Fallback (no reviews): "Hi [first name], I noticed [business name] doesn't have a website yet."
```

Cost: ~$0.001 per lead. At 50 leads: $0.05 total. Negligible.
Model: `claude-haiku-4-5-20251001` (fastest + cheapest for this task).
Error handling: if Claude API call fails (timeout, auth error), fall back to generic
template silently. Log error to console. Never block site generation for a failed
personalization.

Full outreach template output by generate.ts:
```
[personalized first line]
I built you a site — take a look: [site_url]

$99 one-time setup · $9/mo hosting.
To go live: [stripe_payment_link]

Reply STOP to opt out.
```

### 4. Stripe payment link per lead

In `generate.ts`, call Stripe API to create a Payment Link:
- Product: "Website Setup" — one-time $99
- Metadata: `{ lead_slug: "[slug]", business_name: "[name]" }` — used by webhook
- Link expires: never (Payment Links are permanent until archived)
- Store link URL in CSV row and in outreach template

Stripe API key from `STRIPE_SECRET_KEY` env var (add to `.env`).

After $99 payment confirmed, operator manually sets up Stripe subscription ($9/mo) —
automate this in Phase A via webhook.

### 5. Stripe webhook → auto-HITL approval (Phase A only)

**Note:** Manual approval (SQL UPDATE) is fine for Phase C (5-20 leads). The webhook
auto-approval is a Phase A item. Flag it here for planning:

When built: Vercel Edge Function at `/api/stripe-webhook` receives
`checkout.session.completed` event. Extracts `lead_slug` from metadata. Sets
`approved = true` in Supabase. Idempotency: check if `approved` is already `true`
before updating (handles Stripe webhook retries).

### 6. Google Places API as primary scraper (Phase A)

**Phase C:** Operator runs `generate.ts` with a Google Maps URL. Playwright scrapes it
(existing `scraper.ts`). If Playwright fails or returns empty data, generate.ts prompts
the operator for each missing field interactively via `readline` CLI prompts.
**Phase A:** Swap primary source to Places API. `GOOGLE_PLACES_API_KEY` env var.

Places API quota: 1,000 requests/day free tier. At 50 leads/day = 50 requests. Well within
free tier. Rate limit: 10 requests/second. No throttling needed at Phase A volume.

Reviews: Places API returns up to 5 reviews per place. Sufficient.
Fallback: if Places API returns no phone/address, fall back to Playwright scrape.

## Outreach Send Mechanism (Phase C)

**How messages are sent in Phase C:**
1. `generate.ts` prints the formatted WhatsApp message to stdout (already implemented)
2. Operator copies the message and sends manually from personal WhatsApp
3. Operator updates `status` column in `leads-tracker.csv` from `outreach_ready` to
   `outreach_sent` after sending

**No automated sending in Phase C.** This is intentional — test the message quality and
response rate before automating.
